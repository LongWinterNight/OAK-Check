# 11 — Database Administrator

## Scope в Plan 12

**Прямых задач по адаптиву UI нет.** DBA включается, только если возникнет одна из ситуаций ниже.

## Триггеры вовлечения

### 1. SRE (10) нашёл slow queries на мобильных сценариях

Если конкретный мобильный flow тянет тяжёлый запрос (например, drill-down загружает больше данных, чем десктоп) — понадобится:
- Индекс, специфичный для мобильного запроса
- EXPLAIN ANALYZE, при необходимости — частичный индекс или covering index

Это всё по ранее выполненному Plan 02 (индексы уже частично заложены).

### 2. RUM-метрики (10) ведём в БД

Если решим хранить Web Vitals в таблице:
- [ ] Согласовать схему `WebVitalMetric` (user_id nullable, url, metric_type enum, value float, ua text, timestamp)
- [ ] Партиционирование по дате (если объём > 100k/день — иначе без)
- [ ] TTL: удалять записи старше 90 дней (политика retention)

Но — **не делаем превентивно**. Пока SRE пишет в логи, БД не нужна.

### 3. Больше трафика на SSE от мобильных клиентов с переподписками

Plan 11 уже закрыл SSE cleanup. Если мобильный reconnect (Full-stack 07) создаёт дополнительные коннекты — мониторить `pg_stat_activity` на предмет открытых коннектов. Проблема больше не DBA, а SRE/Backend.

## Заметки на будущее (Plan 13+)

- **Push-уведомления о дедлайнах** — отдельная таблица subscriptions (endpoint, p256dh, auth keys)
